# What's actually happening inside GridWatch

A plain-English tour of every data structure you can see on screen, and
the role it plays in the Emergency Dispatch Simulator.

## What you're looking at

GridWatch simulates a small city laid out as a grid. Streets run
north–south and east–west; intersections are the nodes; one cell ≈ a
city block.

Three kinds of stations are sprinkled around the map:

- **H** — Hospital (dispatches **A**mbulances)
- **F** — Fire station (dispatches **F**ire trucks)
- **P** — Police station (dispatches **P**olice cars)

Every couple of seconds, an emergency pops up at a random intersection:

| Glyph / icon | Meaning | Severity |
|---|---|---|
| Yellow `!` | Low priority | 1 |
| Orange `!` | Medium priority | 2 |
| Red blinking `!` | High priority — life-threatening | 3 |

The dispatcher's job is to:

1. Notice the new incident.
2. Find the **closest available unit of the right kind** (a fire is no
   help if a heart attack is across town).
3. Compute the **fastest route** along the road network.
4. Send the unit, watch it travel, mark it on-scene, then return to
   station.

Everything you see — the queueing, the search, the map, the leaderboards,
the radio log — is powered by a different data structure from the
syllabus. The point of the simulator is to make those structures
*visible*: while it's running, real Dijkstra calls and real quadtree
queries are firing dozens of times per second.

## Unit 1 — Advanced trees

### Fibonacci heap → **routing**
Every time the dispatcher chooses where to send a unit, it runs
Dijkstra's algorithm on the road graph. The priority queue inside that
Dijkstra is a Fibonacci heap with `decrease_key`, which is what makes
the call amortized `O(E + V log V)` instead of `O((V+E) log V)`. You
can see the running count in the **PQ ops** stat — it ticks up by tens
or hundreds every dispatch.

### AVL tree → **unit roster**
Each ambulance/fire-truck/cop-car is registered in an AVL tree keyed by
unit id. Lookups during dispatch are `O(log n)` — and AVL keeps the
roster balanced no matter the order units are added or removed.

### Red-Black tree → **pending incidents by spawn time**
When dispatch can't immediately assign a unit (everyone busy), the
incident lands in an RB tree ordered by when it was created — useful
for showing how long someone has been waiting.

### Splay tree → **recently-dispatched cache**
Splay trees pull recently-accessed keys to the root. We use one as an
LRU-style cache so the radio log can quickly cite the unit that just
got dispatched without walking the full roster.

### B+ tree → **incident log index**
Every incident, once resolved, is filed into a B+ tree keyed by id.
B+ trees keep their leaves linked, so we can run *range scans* (e.g.,
"give me all incidents 100..200") in linear time after one `O(log n)`
locate.

### Threaded BST → **stack-free in-order replay**
A threaded BST mirrors the pending-incident set. Its right-thread
pointers let us walk every pending incident in chronological order
*without recursion or a stack*. Useful when memory is at a premium —
no call-stack depth, no auxiliary array.

### Huffman tree → **radio-log compression**
The radio log is plain ASCII (lines like `"[t=23.1] AMB-03 dispatched
to medical@Oak&Maple sev=2"`). Real dispatch systems compress these
streams. We build a Huffman tree from the byte frequencies and report
the compression ratio live in the HUD (**Huffman**) — you'll watch it
hover around `0.55–0.65` once the log has interesting structure.

## Unit 2 — Priority queues

The Fibonacci heap above is the *workhorse*. The other heaps are real
implementations the simulator builds and exercises so they're not just
dead code:

### Double-ended priority queue → **severity-ordered pending**
When the dispatcher has more incidents than units, the pending pool is
a DEPQ keyed by severity. `pop_max` returns the most life-threatening
incident first; `pop_min` lets us age out trivial cases for status
checks. This is the structure that makes "always serve the heart attack
before the cat in the tree" work.

### Binomial / Leftist / Skew / Pairing heaps
All five meldable heap variants are present. They're warmed up on
startup and available to swap in if you ever want to benchmark Dijkstra
against alternative priority queues — which is a fun one-line code
change because every heap exposes `push` / `pop_min` / `peek_min`.

## Unit 3 — String data structures

### Trie → **autocomplete**
Type `Oa` into the search box and the dropdown that pops up was
generated by a trie walk: descend to the prefix node, then DFS gathering
every leaf. `O(prefix length + matches)`.

### Compressed (Radix) trie → **dictionary footprint**
The same words live in a radix-compressed trie too — single-child chains
collapsed into edge labels. Same correctness, fewer allocations.

### Suffix array (with Kasai LCP) → **radio-log search**
The radio log is a single growing string. Every 32 lines we rebuild a
suffix array over the whole buffer, which lets us answer
"how many times has `AMB-03` shown up?" in `O(m log n)` per query —
much faster than scanning the log by hand.

### DAWG → **minimal automaton dictionary**
Sort the street names, feed them in, and the DAWG collapses every
shared *suffix* across words. We expose the resulting state count so
you can see how much smaller it is than the raw trie node count.

### Fuzzy dictionary (BK-tree) → **misspelling-tolerant search**
Toggle the `Fuzzy` checkbox in the search box and edit-distance kicks
in. `Maplle` finds `Maple` within two edits. The BK-tree prunes the
search using the triangle inequality, so it doesn't have to compute
edit distance against every word in the dictionary.

## Unit 4 — Randomized

### Skip list → **ETA leaderboard**
Every active unit reports an estimated time of arrival; we keep them
in a skip list ordered by ETA. The HUD's "next unit on scene" banner
is driven by `top(1)`. Skip lists give us `O(log n)` ordered ops with
much simpler code than a balanced tree.

### Treap → **per-station workload ranking**
Stations are ordered by current outstanding workload in a treap.
Random priorities keep the tree balanced in expectation, with split /
merge as the only required operations.

## Unit 5 — Spatial data structures

This is the unit the simulator leans on hardest. *Every* dispatch
involves at least one spatial query.

### Quadtree → **nearest idle unit (the fast path)**
When an incident pops, the dispatcher first calls
`sp_quad_nearest(k=8)` — give me the eight idle units that are closest
in straight-line distance. Then we run Dijkstra on those eight to find
the one with the shortest road-distance. Without the quadtree we'd run
Dijkstra against every single unit; with it, we cut that to a constant
`k`.

### KD-tree → **alternate spatial backup**
A KD-tree of the same points is rebuilt every few seconds. Same role,
different implementation — useful for benchmarking and sanity-checking
the quadtree.

### R-tree → **station coverage rectangles**
Each station has a coverage box. The R-tree indexes those boxes so we
can ask "which stations claim this incident's location?" with a single
range query.

### Segment tree → **rolling 60-second incident counter**
A 60-bucket segment tree gives the HUD's "incidents in the last
N seconds" reading. Fast point updates (a new incident bumps the
current bucket), fast range sums (give me the last 30 seconds).

### Interval tree → **shift schedules**
Each unit has on-duty / off-duty intervals. A stabbing query at "now"
returns every unit currently on shift.

### Range tree → **viewport queries**
A 2D range tree is built once over all points. The web UI uses it for
"what's inside this bounding rectangle" — useful when you want to
filter to a neighbourhood instead of the whole map.

### BSP tree → **static partition of stations**
A BSP tree partitions the static station locations and answers
"which partition does this point fall into" — analogous to how 3D
engines route visibility queries.

## Unit 6 — Miscellaneous

### Disjoint Set Union → **road components**
Every road that *isn't* under construction is unioned into a DSU. The
result tells us how many connected components the city has. If a
blocked-road pattern accidentally splits the city, you'll see the
**comp** stat jump from `1` to `2+` and dispatches to the disconnected
half will fail. (This sometimes happens with small grids — try a
larger one.)

### Persistent list → **event history / undo**
Every state-changing event (incident spawned, unit dispatched, unit
arrived) is pushed onto a persistent linked list. Each push returns a
*new* head while leaving older versions untouched, which gives us an
immutable audit trail with structural sharing — exactly the kind of
"persistent data structure" Unit 6 talks about.

### Succinct bit vector → **compact "has incident here?" map**
For every intersection we keep one bit: is there currently an unresolved
incident here? Storing that in a packed bit vector with `rank1` /
`select1` indices lets us answer "how many incidents in the first half
of the map?" in `O(1)` after a one-time `O(n)` build.

## Why bother building this, exactly

Every textbook chapter on advanced data structures asks you to take it
on faith that "Fibonacci heaps make Dijkstra faster" or "quadtrees beat
linear scans". GridWatch is what those statements look like in motion:

- pause the simulator and you'll see the **PQ ops** counter halt; resume
  and watch it spike during dispatches.
- crank up the spawn rate and the **avg response time** climbs because
  the dispatcher has to fall back to the DEPQ pending pool.
- fuzz the street name with a typo and you get a hit because the
  BK-tree found a neighbour within edit-distance 2.

Each statistic on the HUD is a real measurement of a real algorithm
running on real data — not a static demo. That's the whole point.
